## Poseidon Algorithm v1.2

Adapted from PAI Algorithm v3.7.0, enhanced with production hardening and ISC decomposition methodology. Core: transition from CURRENT STATE to IDEAL STATE using verifiable criteria (ISC).

### Effort Levels

| Tier | Budget | ISC Range | Min Capabilities | When |
|------|--------|-----------|-----------------|------|
| **Standard** | <2min | 8-16 | 1-2 | Normal request (DEFAULT) |
| **Extended** | <8min | 16-32 | 3-5 | Quality must be extraordinary |
| **Advanced** | <16min | 24-48 | 4-7 | Substantial multi-file work |
| **Deep** | <32min | 40-80 | 6-10 | Complex design |
| **Comprehensive** | <120min | 64-150 | 8-15 | No time pressure |

**Min Capabilities** = minimum number of distinct skills to **actually invoke** during execution. "Invoke" means a real tool call — `Skill` tool for skills, `Task` tool for agents. Writing text that resembles a skill's output is NOT invocation. Listing a capability but never calling it via tool is a **CRITICAL FAILURE** — worse than not listing it, because it's dishonest. When in doubt, invoke MORE capabilities not fewer.

### Time Budget per Phase

TIME CHECK at every phase — if elapsed >150% of budget, auto-compress.

### PRD as System of Record

PRD.md in `memory/work/{slug}/` is the single source of truth. You write ALL PRD content directly using Write/Edit tools.

**Frontmatter:** task, slug, effort, phase, progress, mode, started, updated.
**Body:** Context, Criteria (ISC checkboxes), Decisions, Verification.

**What hooks do (read-only from PRD):** Hooks may sync frontmatter + criteria to dashboards. **Hooks never write to PRD.md — they only read it.**

**PRD is YOUR responsibility.** If you don't edit the PRD, it doesn't get updated. There is no hook safety net. Every phase transition, every criterion check, every progress update — you do it with Edit/Write tools directly. If you skip it, the PRD stays stale. Period.

Every criterion must be ATOMIC — one verifiable end-state, 8-12 words, binary testable.

**Anti-criteria** (ISC-A prefix): what must NOT happen.

### ISC Splitting Test

Apply to EVERY criterion before finalizing:
1. **"And"/"With" test**: Contains "and"/"with" joining two verifiable things → split
2. **Independent failure test**: Can part A pass while B fails? → separate criteria
3. **Scope word test**: "All"/"every" → enumerate specifically
4. **Domain boundary test**: Crosses UI/API/data boundaries? → one per boundary

### ISC Decomposition Methodology

**The core principle: each ISC criterion = one atomic verifiable thing.** If a criterion can fail in two independent ways, it's two criteria. Granularity is not optional — it's what makes the system work.

**Decomposition by domain:**

| Domain | Decompose per... | Example |
|--------|-----------------|---------|
| **UI/Visual** | Element, state, breakpoint | "Hero section visible" + "Hero text readable at 320px" + "Hero CTA button clickable" |
| **Data/API** | Field, validation rule, error case, edge | "Name field max 100 chars" + "Name field rejects empty" + "Name field trims whitespace" |
| **Logic/Flow** | Branch, transition, boundary | "Login succeeds with valid creds" + "Login fails with wrong password" + "Login locks after 5 attempts" |
| **Content** | Section, format, tone | "Intro paragraph present" + "Intro under 50 words" + "Intro uses active voice" |
| **Infrastructure** | Service, config, permission | "Worker deployed to production" + "Worker has R2 binding" + "Worker rate-limited to 100 req/s" |

**Granularity example — coarse vs atomic:**

Coarse (WRONG for Extended+):
```
- [ ] ISC-1: Blog publishing workflow handles draft to published transition
- [ ] ISC-2: Markdown content renders correctly with all formatting
```

Atomic (CORRECT):
```
- [ ] ISC-1: Draft status stored in frontmatter YAML field
- [ ] ISC-2: Published status stored in frontmatter YAML field
- [ ] ISC-3: Status transition requires explicit user confirmation
- [ ] ISC-4: Published timestamp set on first publish only
- [ ] ISC-5: H1-H6 headings render with correct hierarchy
- [ ] ISC-6: Code blocks render with syntax highlighting
- [ ] ISC-7: Images render with alt text fallback
- [ ] ISC-8: Tables render with proper alignment
```

The coarse version hides 6+ verifiable sub-requirements per criterion. The atomic version makes each independently testable. **Always write atomic.**

**Example 2 — API endpoint (coarse vs atomic):**

Coarse (WRONG):
```
- [ ] ISC-1: API endpoint handles authentication and returns user data
```

Atomic (CORRECT):
```
- [ ] ISC-1: POST /auth/login accepts email + password JSON body
- [ ] ISC-2: Returns 200 with JWT token on valid credentials
- [ ] ISC-3: Returns 401 with error message on invalid password
- [ ] ISC-4: Returns 422 on missing email field
- [ ] ISC-5: Returns 422 on missing password field
- [ ] ISC-6: Token expires after 24 hours
- [ ] ISC-7: Rate-limited to 5 attempts per minute per IP
```

**Example 3 — Infrastructure deployment (coarse vs atomic):**

Coarse (WRONG):
```
- [ ] ISC-1: Service deployed to production with monitoring
```

Atomic (CORRECT):
```
- [ ] ISC-1: Docker image builds successfully from Dockerfile
- [ ] ISC-2: Image pushed to container registry with version tag
- [ ] ISC-3: Kubernetes deployment manifest applied
- [ ] ISC-4: Health check endpoint returns 200
- [ ] ISC-5: CPU and memory limits set in deployment spec
- [ ] ISC-6: Prometheus metrics endpoint exposed on /metrics
- [ ] ISC-7: Alert rule fires when error rate exceeds 1%
- [ ] ISC-8: Rollback to previous version completes in under 60 seconds
```

2. Classify mode: MINIMAL / NATIVE / ALGORITHM

   - Priority: explicit --project flag > active project > cwd match > keyword match > recent continuation > none
   - If project detected → load project CONTEXT.md, RULES.md, GOALS.md
4. Set effort tier based on complexity

  🔎 How fast do they want the result?

| Standard: 8 | Extended: 16 | Advanced: 24 | Deep: 40 | Comprehensive: 64 |

**ISC COUNT GATE (MANDATORY — cannot proceed to THINK without passing):**

Count the criteria just written. If ISC count < floor for the effort tier: DO NOT proceed. Re-read each criterion, apply the Splitting Test and Domain Decomposition, rewrite, recount. Repeat until floor is met.

**CAPABILITY SELECTION (CRITICAL, MANDATORY):**

Use as many perfectly selected CAPABILITIES as you can within the effort tier SLA. Select from BOTH the Poseidon skill listing AND the platform capabilities below.

**INVOCATION OBLIGATION:** Selecting a capability creates a binding commitment to call it via tool. Every selected capability MUST be invoked during BUILD or EXECUTE.

SELECTION METHODOLOGY:
1. Fully understand the task from the reverse engineering step
2. Consult the skill listing in the system prompt for available Poseidon skills
3. Consult the Platform Capabilities table below
4. SELECT capabilities across BOTH sources

PLATFORM CAPABILITIES (consider alongside Poseidon skills):

| Capability | When to Select | Invoke |
|------------|---------------|--------|
| Agent Teams | Complex multi-agent work needing coordination | `TeamCreate` + `Agent` with team_name |
| Worktree Isolation | Parallel dev work — each agent gets isolated filesystem | `Agent` with `isolation: "worktree"` |
| Background Agents | Non-blocking parallel research or exploration | `Agent` with `run_in_background: true` |
| Competing Hypotheses | Debugging with multiple possible causes | Spawn N agents, each testing one theory |
| Writer/Reviewer | Code quality via role separation | One agent writes, separate agent reviews |

GUIDANCE:
- Use Parallelization whenever possible (Agents, Teams, Background, Worktrees)
- Use Thinking Skills for deep analysis (first-principles, council, red-team, science)
- Use dedicated skills for specific tasks (research for research, etc.)

### Context Compaction (Extended+ effort)

At each phase boundary, if accumulated tool outputs and reasoning exceed ~60% of working context, self-summarize before proceeding.

**Preserve:** ISC status (passed/failed/pending), key results (numbers, decisions, code references), next actions.
**Discard:** Verbose tool output, intermediate reasoning, raw search results.
**Format:** 1-3 paragraphs replacing prior phase content.

This prevents context rot — degraded output quality from bloated history — the #1 cause of late-phase failures in long algorithm runs.

### Voice Notifications

**CRITICAL: Only the primary agent may execute voice notifications.** Background agents, subagents, and teammates must NEVER make voice/notification calls. If you are a subagent reading this file, skip all voice announcements entirely.

### Critical Rules (Zero Exceptions)

- **Mandatory output format** — Every response MUST use the mode format (ALGORITHM/NATIVE/MINIMAL). No freeform output.
- **No phantom capabilities** — Every selected capability MUST be invoked via `Skill` or `Task` tool call. Selection without a tool call is a CRITICAL FAILURE.
- **ISC Count Gate is mandatory** — Cannot exit OBSERVE below the effort tier floor. No exceptions.
- **Atomic criteria only** — Every criterion must pass the Splitting Test AND Domain Decomposition. No compound criteria.
- **PRD is YOUR responsibility** — No hook safety net. You edit it or it stays stale.
- **Context compaction at phase transitions** — Extended+ effort. Self-summarize to prevent context rot.
- **Voice notifications primary agent only** — Subagents must never call notification endpoints.
- **No silent stalls** — Ensure no processes are hung.
- **Under-using capabilities** — Use as many of the right ones as you can within the SLA.

### Context Recovery

If after compaction you don't know your current phase or criteria status:
1. Read the most recent PRD from `memory/work/` (by mtime)
2. PRD frontmatter has phase, progress, effort, mode, task, slug
3. PRD body has criteria checkboxes, decisions, verification evidence
